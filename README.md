## 📊 Model Performance

| Metric | Score |
|--------|-------|
| ROC-AUC | **0.999** |
| PR-AUC  | **0.87**  |
| Precision (fraud) | ~0.93 |
| Recall (fraud)    | ~0.82 |

- **Algorithm**: XGBoost (300 estimators, max depth 6, lr 0.05)
- **Imbalance handling**: SMOTE oversampling (0.17% fraud rate)
- **Explainability**: SHAP TreeExplainer — per-prediction feature attribution

## ✨ Key Features

| Feature | Description |
|---|---|
| **SHAP Explainability** | Every prediction shows exactly which features drove it — red = pushed toward fraud, green = pushed toward legitimate |
| **Live Transaction Stream** | Simulates real-time transactions with realistic 98/2 fraud distribution |
| **Risk Timeline** | Live Plotly chart of fraud probability over time with danger zone shading |
| **Risk Gauge** | Color-coded probability meter — green / amber / red zones |
| **Transaction Log** | Scrollable feed of last 200 transactions, exportable as CSV |
| **SMOTE** | Synthetic minority oversampling handles extreme class imbalance properly |
| **Model Stats Endpoint** | ROC-AUC, PR-AUC, confusion matrix all accessible via `/stats` |
| **Input Validation** | Pydantic validators reject bad inputs (e.g. negative Amount) with clear errors |
| **Health Endpoint** | `/health` used by Docker healthcheck to gate frontend startup |
| **Dockerized** | Full `docker-compose` stack — one command to run everything |

## 🚀 Quick Start

### Option A — Docker (Recommended)

```bash
# 1. Get the dataset
#    https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud
#    Place creditcard.csv inside data/

# 2. Train the model first (required before Docker build)
cd backend
pip install -r requirements.txt
python train.py                  # ~2 min — saves artifacts to model/

# 3. Launch full stack
cd ..
docker-compose up --build
```

Open **http://localhost:8501** 🎉

---

### Option B — Local (No Docker)

```bash
# Terminal 1 — Backend
cd backend
python -m venv venv && venv\Scripts\activate   # Windows
# source venv/bin/activate                     # Mac/Linux
pip install -r requirements.txt
python train.py                                # run once
uvicorn main:app --reload --port 8000

# Terminal 2 — Frontend
cd frontend
python -m venv venv && venv\Scripts\activate
pip install -r requirements.txt
streamlit run app.py
```

## 🔌 API Reference

### `POST /predict`

Request body:
```json
{
  "Time": 406.0,
  "Amount": 149.62,
  "V1": -1.359807,
  "V2": -0.072781,
  "...": "...",
  "V28": 0.021476
}
```

Response:
```json
{
  "prediction": 1,
  "probability": 0.9821,
  "risk_level": "HIGH",
  "shap_values": [-0.12, 0.34, "..."],
  "top_features": [
    { "feature": "V14", "shap_value": 0.8234 },
    { "feature": "V4",  "shap_value": -0.4123 }
  ],
  "timestamp": "2024-01-15T10:30:00"
}
```

### `GET /health`
```json
{ "status": "ok", "model_loaded": true, "version": "2.0.0" }
```

### `GET /stats`
Returns full evaluation report: ROC-AUC, PR-AUC, confusion matrix, precision/recall per class.

Interactive docs available at **http://localhost:8000/docs**

## 📈 Design Decisions

**Why XGBoost over Logistic Regression / Random Forest?**
Gradient boosting handles the non-linear feature interactions in PCA-transformed data better, and is fast enough for real-time inference (<50ms per prediction).

**Why SMOTE over class weighting?**
Class weighting just penalizes misclassification. SMOTE generates synthetic minority samples, giving the model more signal to learn fraud patterns — leading to significantly better recall.

**Why SHAP over built-in feature importance?**
Global feature importance tells you which features matter on average across the dataset. SHAP gives per-transaction attribution — you can see *why this specific transaction* was flagged. This is a real compliance requirement in production fintech systems.

**Why PR-AUC alongside ROC-AUC?**
With 0.17% fraud, a model predicting "never fraud" achieves 99.83% accuracy and a decent ROC-AUC. PR-AUC penalizes this behavior and is a much more honest metric for imbalanced classification.
